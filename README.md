資料來源：[農業部農業資料開放平臺－農產品交易行情](https://data.moa.gov.tw/open_detail.aspx?id=037)

## 快速開始

環境需求：Python 3.10 以上，不需安裝額外套件。

先抓一頁資料確認 API 可用：

```powershell
python fetch_agri_prices.py --max-pages 1 --csv data/sample.csv
```

抓指定日期、作物與市場：

```powershell
python fetch_agri_prices.py `
  --start 2026-06-01 `
  --end 2026-06-15 `
  --crop "甘藍-初秋" `
  --market "台北一" `
  --csv data/cabbage_taipei.csv
```

回補五個核心品項近兩年資料並重新執行稽核：

```powershell
powershell -ExecutionPolicy Bypass -File .\backfill_core_crops.ps1
```

只執行資料充足度稽核：

```powershell
python audit_data_readiness.py
```

產生模型可直接使用的每日全市場加權平均資料：

```powershell
python prepare_model_dataset.py
```

會產生兩份資料：

- `data/model/core_xiluo_prices.csv`：五個核心品項在西螺鎮市場的固定市場時間序列，建議作為主要預測資料。
- `data/model/core_daily_prices.csv`：每日全市場加權平均資料，建議用於 Dashboard 與輔助分析。

全市場彙總資料每列代表一個核心品項在一個交易日的彙總結果，包含：

| 欄位 | 定義 |
| --- | --- |
| `trade_date` | 西元交易日期 |
| `crop_code`、`crop_name` | 作物代號與名稱 |
| `weighted_average_price` | 依各市場交易量加權計算的每日平均價格 |
| `total_volume` | 當日所有市場總交易量 |
| `market_count` | 當日有交易資料的市場數 |
| `minimum_market_price` | 當日各市場平均價中的最低值 |
| `maximum_market_price` | 當日各市場平均價中的最高值 |

部分日期只有少數市場有交易，因此全市場加權平均可能受到市場組成變化影響。正式模型應優先使用固定市場時間序列，或將 `market_count` 設為品質門檻。

省略篩選條件時，API 預設回傳查詢日曆日近四天（含當天）的資料。程式會以每頁 1,000 筆搭配 `$skip` 自動翻頁，避免只抓到第一頁。

## 輸出

- `data/agri_prices.sqlite3`：累積寫入的 SQLite 資料庫
- `--csv` 指定的檔案：僅包含該次抓取結果，使用 Excel 可辨識的 UTF-8 編碼

SQLite 資料表名稱為 `market_prices`。交易日期會同時保存 API 原始民國日期 `trade_date_roc`，以及方便分析的西元日期 `trade_date`。

## 資料欄位說明

| 欄位 | 代表意思 | 範例 |
| --- | --- | --- |
| `交易日期` | 批發市場的交易日期，使用民國年格式 `YYY.MM.DD` | `115.06.12` 代表西元 `2026-06-12` |
| `種類代碼` | 農產品所屬種類的分類代碼 | `N04`、`N05`、`N06` |
| `作物代號` | 農產品品項與品種的識別代碼 | `LA1`、`FA810` |
| `作物名稱` | 農產品名稱，可能包含品種或規格 | `甘藍-初秋`、`康乃馨-多朵紅` |
| `市場代號` | 批發市場的識別代碼 | `104` |
| `市場名稱` | 發生交易的批發市場名稱 | `台北二`、`西螺鎮` |
| `上價` | 當日價格較高的一組交易之平均價格 | `50.0` 元／公斤 |
| `中價` | 當日價格中間區段交易之平均價格 | `40.0` 元／公斤 |
| `下價` | 當日價格較低的一組交易之平均價格 | `30.0` 元／公斤 |
| `平均價` | 該作物在該市場當日的整體平均交易價格 | `39.8` 元／公斤 |
| `交易量` | 該作物在該市場當日的總交易重量 | `1250.0` 公斤 |

### 價格欄位差異

`上價`、`中價`、`下價`不是當日最高價、中位數與最低價，而是將交易依價格排序後，分成不同交易量區段計算的代表價格。

進行趨勢分析或價格預測時，通常可以使用：

- `平均價`：作為主要預測目標。
- `交易量`：判斷市場供需狀況。
- `上價 - 下價`：衡量當日價格分散程度。
- `市場名稱`：比較不同市場價格。
- `作物代號`：比 `作物名稱` 更適合當作資料識別欄位。

若資料的 `作物名稱` 為「休市」，價格與交易量通常都是 `0`，分析時應排除或獨立標記。

### 產銷履歷與食品安全資料

產銷履歷、有機驗證與檢驗結果較適合用於採購推薦與供應來源篩選，不建議直接作為每日價格預測的主要特徵。

採購推薦分數可以設計為：

```text
採購推薦分數
= 價格穩定分數
+ 交易量分數
+ 產銷履歷分數
+ 有機驗證分數
+ 檢驗合格分數
```

這能讓系統推薦價格合理、來源可追溯且品質風險較低的農產品。

### 暫時不直接納入第一版模型

| 資料集 | 原因 |
| --- | --- |
| 農業部自動氣象站與雨量站資料 | 可取得即時資料，但實測指定歷史日期回傳空資料，暫時無法與兩年價格資料完整對齊 |
| 農藥與肥料完整清單 | 偏向靜態知識資料，與每日市場價格的直接關聯較弱 |
| 產銷履歷詳細栽種流程 | 資料結構複雜，第一版處理成本較高 |
| 肉類行情 | 資料粒度與價格形成方式不同，應使用獨立 Dashboard 與模型 |

### 建議的正式專題範圍

第一版正式納入：

1. 農產品交易行情：建立核心價格模型。
2. 市場休市日：執行資料品質檢查。
3. 植物疫情與害蟲旬報：解釋價格異常與供應風險。
4. 產銷履歷與檢驗結果：建立安全採購推薦。
5. 生產成本：進行價格與成本分析。

氣象、肉類及有機驗證可作為第二階段延伸展示。此範圍可完整呈現：

資料取得
→ 資料品質檢查
→ 價格預測

## 是否將肉類納入專題

第一版建議將蔬果作為核心範圍，肉類行情只做延伸展示，不直接放入同一個價格預測模型。

原因如下：

- 蔬果以「公斤、作物、市場」為主要粒度；毛豬以「頭數、平均重量、重量級距」為主要粒度，資料結構不同。
- 白肉雞與雞蛋行情的市場粒度與蔬果不同，需要獨立的清洗方式與模型。
- 肉類和蔬果通常不是直接替代品，若一起做採購推薦，需要先定義營養、菜單與預算限制。
- 同時建立蔬果、毛豬與家禽模型會明顯增加第一版的開發與驗證範圍。

建議展示方式：

1. 核心功能使用蔬果行情，完成價格趨勢、異常偵測、預測及採購推薦。
2. Dashboard 增加「其他食材行情」頁面，展示豬肉、雞蛋與白肉雞最新價格。
3. 後續版本再為肉類建立獨立模型，最後整合成團膳採購建議。
